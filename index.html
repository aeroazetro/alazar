1. The Problem at a Glance

Firmware goes directly from supplier to aircraft with no automated adversarial testing in between. Manual red-teaming covers <5% of components. Ransomware attacks on aviation IoT increased 600% between Jan 2024 – Apr 2025 [1]. Airbus was directly breached through a supplier in January 2023. Firmware represents the most critical and least-studied attack surface in embedded systems [2].

3. Innovation #1: Spec-Conformance Oracle

This is our core novelty. Existing fuzzers [3] ask "did it crash?" We ask "did it follow its datasheet?" A firmware that returns subtly wrong navigation data is far more dangerous than one that crashes — and normal fuzzers would miss it entirely.

sequenceDiagram
    participant Spec as I/O Specification
(Component Datasheet)
    participant Fuzz as AI Fuzzer
    participant FW as Firmware Under Test
(Emulated)
    participant Oracle as Spec-Conformance
Oracle

    Spec->>Fuzz: Define valid input space
+ expected output ranges
    Fuzz->>FW: Send adversarial input
(edge-case within spec)
    FW->>Oracle: Return actual output
    Spec->>Oracle: Expected output for this input
    
    alt Output matches spec
        Oracle-->>Oracle: PASS - Behaves as specified
    else Output deviates from spec
        Oracle-->>Oracle: DEVIATION DETECTED
    end
Example: IMU returns heading 274° when spec requires 271°±1° under this input condition. A normal fuzzer sees "no crash" and passes it. Our oracle catches the 3° deviation — which could cause a navigational error.

4. Innovation #2: Cross-Supplier Fingerprinting

Every firmware submission generates a behavioral profile. The pipeline compares profiles across all suppliers for the same component class. Statistical outliers are automatically flagged — the population defines "normal," not the vendor.

5. Innovation #3: Deployment-Context Probing

The AI fuzzer knows the component's actual operational envelope — flight routes, altitude ranges, GPS bounding boxes, maintenance cycles. It prioritizes inputs from within this context to trigger deployment-specific logic bombs. This approach extends directed greybox fuzzing [4] with aerospace-specific operational context.

7. Tiered Access Model

The Black-Box tier requires zero supplier cooperation — testing against public datasheets only. The White-Box tier uses firmware emulation via QEMU with HALucinator-style abstraction [5] and P2IM peripheral modeling [6] for deeper binary analysis using angr [7]. The enclave's hardware attestation [8] makes firmware submission palatable.

10. Regulatory Tailwind — Why Now

timeline
    title Regulatory and Threat Timeline
    2019 : DO-326A / DO-356A adopted as AMC
         : DO-356A names fuzz testing as refutation activity
    2023 : Airbus supplier breach — 3000+ vendors affected
         : Boeing $200M ransomware attack
    2024 : 600% ransomware increase in aviation IoT
         : FAA NPRM — proposed mandatory firmware screening
    2025 : EASA Part-IS — Production Orgs must comply (Oct)
         : FAA committee on cybersecurity standards (May)
    2026 : EASA Part-IS — Operators must comply (Feb)
         : OUR PROPOSAL — FYI 2026 (Mar)
This is our strongest "why now" argument. DO-356A explicitly names fuzz testing as a certification-acceptable "refutation activity" [10]. Two years ago this proposal would have been speculative. Today the regulations are arriving — and no tool exists to help Airbus comply.
